<task type="auto">
  <name>Task 1: Add PID lockfile to prevent concurrent scheduler runs</name>
  <files>bot/scheduler.py</files>
  <action>
Add a PID-based lockfile mechanism to `run_pipeline()` in scheduler.py. The lock must:

1. Define a lock path: `Path(config.bot.logs_dir) / "scheduler.lock"`
2. At the start of `run_pipeline()`, check if the lockfile exists:
   - If it exists, read the PID from it
   - Check if that PID is still alive (`os.kill(pid, 0)` on Unix, or `psutil` — but avoid new deps, use `os.kill` with try/except for cross-platform)
   - If alive: log a warning "Another scheduler instance (PID {pid}) is running — exiting" and return False
   - If dead (stale lock): log info, delete the stale lockfile, proceed
3. Write current PID (`os.getpid()`) to the lockfile
4. Use a try/finally to ensure the lockfile is deleted when `run_pipeline()` exits (success or failure)
5. Add `import os` at the top if not already imported

Cross-platform PID check approach — avoid psutil dependency:
```python
def _is_pid_alive(pid: int) -> bool:
    try:
        os.kill(pid, 0)
        return True
    except (OSError, ProcessLookupError):
        return False
```
On Windows, os.kill(pid, 0) raises OSError if the process doesn't exist. On Unix, it raises ProcessLookupError or OSError. Both are handled.

Do NOT use `fcntl` or `msvcrt` — keep it simple with PID file approach.

2. Confirm the lock path uses config.bot.logs_dir
3. Confirm try/finally ensures cleanup
4. Run `python -c "import ast; ast.parse(open('bot/scheduler.py').read())"` to verify syntax
  </verify>
  <done>
run_pipeline() checks for and creates a PID lockfile. If another instance is running, it logs a warning and returns False. Lockfile is always cleaned up via try/finally.
